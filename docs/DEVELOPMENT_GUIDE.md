# AfterSaleAgent Development Guide

## 1. Project Positioning

AfterSaleAgent is a B2B ecommerce after-sales agent backend system.

The target user is not an end consumer. The target user is a merchant support team member who works inside an internal customer service dashboard.

The system helps support agents handle requests such as:

- refund
- return and refund
- shipment issue
- missing item
- compensation
- after-sales ticket creation
- high-risk refund approval

The project should demonstrate both backend engineering ability and agent application ability.

The most important engineering signals are:

- clear FastAPI API design
- relational database modeling
- async database access
- document ingestion and vector retrieval
- LangGraph state machine
- tool calling with persisted records
- Server-Sent Events streaming
- human approval for high-risk actions
- audit logging
- testable and reproducible local setup
- clean README and API examples

## 2. Product Summary

### One-line Description

AfterSaleAgent is an intelligent after-sales backend platform for merchant support teams. It can query orders, retrieve policies, assess refund risk, create refund requests or tickets, and route high-risk actions to human approval.

### Example User Story

A support operator receives a customer request:

> The customer says order 10086 was delivered two days ago, but they are not satisfied and want a refund.

The operator sends the message to AfterSaleAgent.

The system should:

1. Create an agent run.
2. Extract intent and order id.
3. Query order, payment, shipment, product, and customer data.
4. Search after-sales policy documents.
5. Decide whether the refund is allowed.
6. Decide whether manual approval is required.
7. Create a refund request, after-sales ticket, or approval.
8. Generate a reply that the operator can send to the customer.
9. Save every step, tool call, decision, and final result.

## 3. Target Roles

Keep the permission model simple in MVP.

### admin

Can manage knowledge base documents, view all orders, view all runs, and view all approvals.

### agent

Can start agent runs, view assigned tickets, and view order information.

### reviewer

Can review high-risk refund approvals.

## 4. MVP Scope

### Must Have

1. User registration and login with JWT.
2. Role-based access checks for basic routes.
3. Mock ecommerce data: customers, products, orders, order items, shipments, payments.
4. After-sales policy document upload.
5. Document chunking, embedding, and pgvector storage.
6. Knowledge search API.
7. LangGraph agent workflow.
8. Tool calling:
   - get_order
   - get_shipment
   - get_payment
   - search_policy
   - create_after_sales_ticket
   - create_refund_request
   - create_approval
9. Agent run creation API.
10. SSE endpoint for streaming run events.
11. Persisted agent_runs, agent_steps, tool_calls, approvals, and audit_logs.
12. Docker Compose local environment.
13. README, demo data, API examples, and tests.

### Nice To Have

1. Lightweight frontend generated by AI.
2. Evaluation dataset.
3. GitHub Actions.
4. OpenTelemetry or Langfuse integration.
5. Admin dashboard metrics.
6. Multi-model provider config.

### Do Not Build In MVP

1. Real Taobao, JD, Shopify, or payment integration.
2. Kubernetes deployment.
3. Complex billing.
4. Full multi-tenant SaaS.
5. Complex frontend before the backend is stable.
6. Autonomous open-ended agent that can call arbitrary tools.

## 10. LangGraph Agent Design

### Principle

Use a controlled state machine instead of an unconstrained autonomous agent.

This is better for backend interviews because you can clearly explain:

- state transition
- tool boundary
- approval condition
- persistence
- failure recovery
- auditability

### Agent State

```python
class AfterSaleAgentState(TypedDict, total=False):
    run_id: str
    user_id: str
    customer_id: str
    message: str
    intent: str
    order_id: str | None
    order: dict | None
    shipment: dict | None
    payment: dict | None
    policy_chunks: list[dict]
    risk_level: str
    requires_approval: bool
    action: str
    tool_calls: list[dict]
    final_answer: str
    error: str | None
```

### Nodes

```text
parse_intent
  -> extract intent, order id, refund reason

load_order
  -> call get_order
  -> call get_payment
  -> call get_shipment

retrieve_policy
  -> call search_policy

assess_risk
  -> apply deterministic business rules
  -> optionally ask LLM to explain the decision

decide_action
  -> choose refund, ticket, approval, reject, or ask for more information

execute_action
  -> create refund request, ticket, or approval

finalize_response
  -> generate support-agent-facing answer
```

### Intent Types


## 12. Business Rules

Start with deterministic rules. Let the LLM help with language and intent, not final authority.

### Refund Rules

```text
If order does not exist:
  ask for more information

If payment status is not paid:
  reject refund

If order status is pending_payment:
  reject refund

If order status is paid and shipment is not_shipped:
  allow direct refund

If shipment is delivered and delivery date is within 7 days:
  ordinary product can be refunded

If product is virtual:
  reject refund by default

If product is fresh and shipment is delivered:
  reject no-reason refund by default

If order total amount >= 500:
  require manual approval

If order is already refunded:
  reject duplicate refund
```

### Approval Rules

```text
Refund amount >= 500:
  create approval

Multiple refund attempts on same order:
  create approval

Shipment status is lost:
  create approval

Customer level is high value and refund is ambiguous:
  create approval
```

## 15. Embedding Strategy

For MVP, choose one of these:

### Option A: Local embedding

Use `sentence-transformers` locally.

Pros:

- stable
- no embedding API cost
- good for demo

Cons:

- larger dependency
- slower on weak machines

### Option B: API embedding

Use an external embedding provider.

Pros:

- simpler runtime performance

Cons:

- more provider dependency

Recommended for you:

Use local embedding first. It makes the project easier to run as an open source demo.

## 21. Interview Talking Points

Use these points when explaining the project:

1. This is not just a chatbot. It is a backend-controlled agent workflow.
2. The agent cannot perform high-risk actions freely. Refunds above a threshold require approval.
3. All agent runs, steps, and tool calls are persisted for observability and debugging.
4. The policy retrieval system uses pgvector, which keeps RAG inside PostgreSQL.
5. LangGraph is used because after-sales workflows need state, branching, and tool boundaries.
6. LLM output is not trusted as final business truth. Deterministic rules are used for key refund decisions.
7. SSE is used to stream long-running agent progress to the frontend.
8. The LLM provider is abstracted so DeepSeek can be replaced later.
