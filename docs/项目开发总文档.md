# AfterSaleAgent 项目开发总文档

这份文档是你开发 AfterSaleAgent 时的主文档。你可以把它当成一个小型 PRD + 技术设计 + 开发计划。

项目目标不是做一个“能聊天的机器人”，而是做一个能体现真实后端能力的 Agent 应用：

- 有清晰业务场景
- 有数据库建模
- 有后端 API
- 有知识库检索
- 有 LangGraph Agent 流程
- 有工具调用
- 有人工审批
- 有审计日志
- 有流式输出
- 有测试和 README

## 1. 项目定位

项目名：`AfterSaleAgent`

项目类型：B 端电商售后 Agent 后端系统。

目标用户：商家的客服人员、售后主管、后台管理员。

一句话介绍：

> AfterSaleAgent 是一个面向商家客服后台的电商售后 Agent 系统，帮助客服自动判断退款、退货、赔付请求，查询订单和物流，检索售后政策，创建工单或退款申请，并对高风险操作走人工审批。

你面试时可以这样介绍：

> 我做的是一个生产风格的 Agent 后端系统，不只是普通聊天。用户输入售后问题后，系统会通过 LangGraph 控制流程，查询订单、检索政策、调用工具、判断风险、创建退款或审批，并把每一步持久化下来，方便审计和排查。

## 2. 典型业务场景

客服收到用户诉求：

```text
订单 10086 已经签收两天了，用户不满意，想申请退款，可以直接退吗？
```

客服把这句话输入系统。

系统自动完成：

1. 创建一次 Agent Run。
2. 识别意图：退款申请。
3. 提取订单号：10086。
4. 查询订单、商品、支付和物流。
5. 检索售后政策知识库。
6. 判断是否符合退款规则。
7. 判断是否需要人工审批。
8. 创建退款申请、售后工单或审批单。
9. 返回一段客服可直接回复用户的话。
10. 保存完整执行记录和审计日志。

## 3. 用户角色

第一版只做 3 个角色，不要做复杂权限系统。

### admin

管理员。

可以：

- 管理知识库文档
- 查看所有订单
- 查看所有 Agent Run
- 查看所有审批
- 查看审计日志

### agent

客服人员。

可以：

- 发起售后 Agent 会话
- 查看订单
- 查看自己创建的工单和退款申请
- 查看 Agent 运行详情

### reviewer

售后主管。

可以：

- 查看待审批退款
- 批准审批
- 拒绝审批

## 4. MVP 功能范围

你第一版必须完成这些：

1. 用户注册、登录、JWT 鉴权。
2. 基础角色权限：`admin`、`agent`、`reviewer`。
3. 模拟电商数据：客户、商品、订单、订单项、物流、支付。
4. 售后政策文档上传。
5. 文档切块、embedding、pgvector 入库。
6. 知识库检索 API。
7. LangGraph 售后 Agent 流程。
8. 工具调用：
   - 查询订单
   - 查询物流
   - 查询支付
   - 检索政策
   - 创建售后工单
   - 创建退款申请
   - 创建审批单
9. Agent Run 创建接口。
10. SSE 流式事件接口。
11. 保存 Agent Run、Step、Tool Call、Approval、Audit Log。
12. Docker Compose 一键启动。
13. 测试、README、演示数据和演示脚本。

第一版不要做这些：

1. 真实淘宝、京东、Shopify 接口。
2. 真实支付退款。
3. 复杂前端。
4. Kubernetes。
5. 多租户计费。
6. 太自由的 autonomous agent。
7. 十几个 Agent 互相协作。

你的目标是：**小而完整，能跑通，能讲清楚。**

## 10. LangGraph Agent 设计

### 核心原则

第一版不要让 Agent 自由乱跑。

你要做的是固定流程的 Agent 状态机：

```text
parse_intent
  -> load_order
  -> retrieve_policy
  -> assess_risk
  -> decide_action
  -> execute_action
  -> finalize_response
```

这样更适合面试，因为你可以讲清楚：

- 每一步做什么
- 状态如何流转
- 什么时候调用工具
- 什么时候需要审批
- 为什么关键业务判断不能完全交给 LLM
- 如何保存执行过程

### Agent State

```python
class AfterSaleAgentState(TypedDict, total=False):
    run_id: str
    user_id: str
    customer_id: str
    message: str
    intent: str
    order_id: str | None
    order: dict | None
    shipment: dict | None
    payment: dict | None
    policy_chunks: list[dict]
    risk_level: str
    requires_approval: bool
    action: str
    tool_calls: list[dict]
    final_answer: str
    error: str | None
```

### 节点说明

`parse_intent`

识别用户意图，提取订单号、退款原因。

`load_order`

调用订单、支付、物流工具，把业务数据加载到 state。

`retrieve_policy`

根据用户问题和订单状态检索售后政策。

`assess_risk`

根据规则判断风险等级。

`decide_action`

决定下一步动作：

```text
create_refund
create_ticket
create_approval
reject_refund
ask_for_more_info
answer_only
```

`execute_action`

真正创建退款申请、工单或审批单。

`finalize_response`

生成客服可读的最终回复。

### 意图类型

```text
refund_request
return_refund_request
shipment_issue
compensation_request
order_status_query
complaint
unknown
```

### 风险等级

```text
low
medium
high
```

## 11. 工具设计

工具调用要有边界，不能只是函数随便调。

每个工具都应该：

1. 校验输入。
2. 执行业务逻辑。
3. 写入 `tool_calls` 表。
4. 返回结构化 JSON。
5. 明确暴露错误。

## 12. 售后业务规则

第一版关键判断尽量用确定性规则，不要完全相信 LLM。

```text
订单不存在：
  要求补充信息

支付状态不是 paid：
  拒绝退款

订单状态是 pending_payment：
  拒绝退款

订单已支付但未发货：
  可直接退款

订单已签收，签收时间 <= 7 天，普通商品：
  可申请退款

虚拟商品：
  默认不可退

生鲜商品已签收：
  默认不可无理由退款

订单金额 >= 500：
  需要人工审批

订单已经退款：
  拒绝重复退款
```

审批规则：

```text
退款金额 >= 500：
  创建审批

同一个订单多次退款：
  创建审批

物流状态为 lost：
  创建审批

高价值客户且规则不明确：
  创建审批
```

## 13. 知识库样例文档

你需要准备 6-8 条售后政策。

建议主题：

1. 7 天无理由退货规则
2. 未发货订单取消规则
3. 已签收订单退款规则
4. 虚拟商品退款规则
5. 生鲜商品退款规则
6. 高金额退款审批规则
7. 物流丢件赔付规则
8. 重复退款防控规则

样例文本：

```text
普通实物商品在签收后 7 天内，商品未使用且不影响二次销售的，可以申请 7 天无理由退货退款。

虚拟商品包括软件授权、兑换码、会员卡、数字内容等。一旦完成交付，除非存在明确服务故障，否则不支持无理由退款。

生鲜商品签收后不支持无理由退货。如果用户反馈质量问题，需要提供照片或视频凭证，由客服创建售后工单处理。

退款金额大于等于 500 元的申请，需要售后主管审批后才能创建正式退款。
```

## 15. Embedding 策略

MVP 推荐先用本地 embedding。

可选方案：

```text
sentence-transformers
```

优点：

- 不依赖额外 API
- 开源项目更容易跑
- 演示更稳定

缺点：

- 依赖稍重
- 本地机器慢一点

第一版不要为了 embedding 供应商纠结。目标是先跑通 RAG 链路。

## 16. SSE 流式事件

接口：

```http
GET /api/v1/agent/runs/{run_id}/events
```

事件类型：

```text
run_started
step_started
step_completed
tool_called
tool_completed
policy_retrieved
risk_assessed
approval_created
refund_created
ticket_created
final_answer
run_failed
run_completed
```

事件 payload：

```json
{
  "run_id": "run_123",
  "event": "tool_called",
  "step": "load_order",
  "data": {
    "tool": "get_order",
    "input": {
      "order_id": "10086"
    }
  },
  "created_at": "2026-06-04T12:00:00Z"
}
```

第一版实现建议：

1. 先在 Agent Run 请求里同步执行 agent。
2. 同时把事件写入数据库。
3. SSE 接口从数据库或内存队列读取。
4. 后面再升级成 Redis stream。

## 17. 统一错误格式

建议所有 API 错误都长这样：

```json
{
  "error": {
    "code": "ORDER_NOT_FOUND",
    "message": "订单不存在",
    "details": {
      "order_id": "10086"
    }
  }
}
```

## 21. 面试讲解点

你可以重点讲这些：

1. 这不是普通 Chatbot，而是可审计的 Agent 后端工作流。
2. LangGraph 用来控制状态流转，避免 Agent 失控。
3. 订单、支付、物流、退款、审批是关系型数据，体现后端建模能力。
4. 售后政策放入 pgvector，体现 RAG 能力。
5. 工具调用全部持久化，便于调试和追踪。
6. 高风险退款不能由 LLM 直接执行，必须走人工审批。
7. SSE 让前端能实时看到 Agent 运行过程。
8. DeepSeek 被封装在 LLMClient 后面，后续可以替换模型供应商。
9. 关键业务判断使用确定性规则，LLM 主要负责理解和表达。
10. Docker Compose 和测试让项目更像真实工程。
